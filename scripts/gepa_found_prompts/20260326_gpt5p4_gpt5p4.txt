1. Output format
- Output exactly one Rust source block wrapped in <code> and </code>.
- Do not include any explanations, comments outside the code, markdown fences, or extra prose.
- The code must compile as a standalone Rust source file in a normal `std` environment.
- Include a minimal `fn main() {}` if needed so the file is a complete standalone binary.
- Preserve crate-level attributes such as `#![allow(...)]` exactly when present.

2. Safety requirements
- The result must contain no `unsafe` blocks, no `unsafe fn`, no raw pointer dereferences, no unions, and no FFI calls.
- Remove `extern "C"` and `#[no_mangle]` whenever necessary; ABI compatibility is not required.
- Replace unsafe constructs with safe Rust equivalents.
- If the original API uses raw pointers, redesign function signatures to use safe references, slices, arrays, owned containers, or return values.
- Never fabricate fake safety by creating unchecked giant slices or relying on unchecked assumptions.

3. Compilation requirements
- The code must be valid Rust syntax and compile without requiring trait imports that are not present.
- Prefer native Rust primitives (`i32`, `u8`, `f32`, etc.) over C aliases unless aliases already exist in the source and are useful.
- Prefer `std` over `core` in all paths. Do not emit `::core::...` paths.
- Do not use `usize::try_from(...)`, `i32::try_from(...)`, or similar trait-based conversions unless you also explicitly import the needed trait. To maximize compilation reliability, prefer simple checked casts such as:
  - `if x < 0 { return; } let xu = x as usize;`
  - and use `checked_mul`, `checked_add`, etc. for overflow-sensitive arithmetic.
- Avoid references to unavailable external symbols like C library math functions; use inherent Rust methods such as `f32::floor()`.

4. Behavioral goals
- Preserve the original behavior as closely as possible within safe Rust.
- Prioritize in this order:
  1) full safety
  2) successful compilation
  3) behavioral fidelity
  4) preserving original signatures/ABI

5. API redesign rules
- Raw input/output pointers should become safe arguments:
  - fixed-size numeric vectors/buffers -> `&[T; N]` / `&mut [T; N]`
  - variable-length buffers -> `&[T]` / `&mut [T]`
  - output-through-pointer structs -> return the struct or accept `&mut Struct`
- If the original struct contains a raw pointer solely for buffer ownership/access, replace it with a safe container such as `Vec<T>` when appropriate.
- If dimensions or counts are stored separately from a vector, clamp processing to the minimum safe length rather than assuming the vector is large enough.

6. Binary parsing and pointer-walking code
- If the original code parses bytes from a raw pointer without a trusted length, redesign the API to accept `&[u8]`.
- Parse using checked indexing and helper functions with `get(...)`, range checks, `checked_add`, and endian-aware integer conversions.
- Reconstruct floats using `f32::from_bits(...)` / `f64::from_bits(...)` after safely reading integer bit patterns.
- Preserve obvious integer error-code conventions if present, or use `Option`/`Result` if that makes the code clearly safe and simpler.

7. Numeric and data-layout guidance
- For byte order conversion, use idiomatic methods like `from_be_bytes`, `from_le_bytes`, `to_be_bytes`, etc.
- For mutable static lookup tables, prefer `const`.
- For math-heavy code, keep the same formula while replacing unsafe indexing/pointer arithmetic with safe indexing and precondition checks.
- When translating pointer arithmetic over strided arrays, infer the logical indexing pattern and express it with safe indexing helpers.

8. Important lessons from prior failures
- Do not emit code that depends on `TryFrom`/`TryInto` unless you import the trait; previous compilations failed specifically because `usize::try_from(...)` was used without `use std::convert::TryFrom;`.
- The safest default is usually to avoid `TryFrom` entirely and instead:
  - check sign manually for signed integers,
  - cast with `as`,
  - and guard arithmetic with `checked_*`.
- Safe fixed-size header access is often best represented as `&[u8; N]`.
- For image/pixel code, a safe redesign may replace `*mut pixel` with `Vec<pixel>` inside the image struct and process only the safely available prefix implied by `w * h`.
- For DSP/math code using pointer offsets into a float buffer, infer the accessed indices and require an input slice long enough for the maximum index used; return early if the slice is too short.
- Keep helper functions private unless the original API clearly needs them public.

9. Standalone compilation expectations
- Ensure every referenced item is defined or imported.
- Ensure boolean-to-integer conversions are written in valid Rust, e.g. `(cond) as i32`.
- If you redesign signatures, make the new signatures plain Rust `pub fn` or `fn`, not `extern "C"`.
